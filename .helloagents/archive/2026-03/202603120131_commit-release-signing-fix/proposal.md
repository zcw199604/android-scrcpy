## 1. 需求

### 背景
当前工作区已经包含一组与“GitHub Release APK 安装失败”修复相关的未提交改动：release 签名配置、workflow 发布策略、README/知识库说明，以及两个未跟踪的 `.helloagents` 归档目录。用户已明确要求把这些**当前未提交内容全部一起提交**，不再单独拆分“签名修复”和“发布归档”两个部分。

### 目标
- 将当前工作区所有未提交改动纳入一次本地提交。
- 提交信息准确概括 release APK 签名修复与发布说明同步。
- 保留本次提交动作的方案记录与执行日志。

### 约束条件
```yaml
时间约束: 本次只创建本地提交，不执行远端推送或新标签发布
兼容性约束: 提交范围按用户确认包含当前全部未提交内容，包括两个 `.helloagents` 归档目录
安全约束: 不改写历史，不执行强制 push 或 tag 覆盖
业务约束: 提交信息需反映“修复 release APK 签名与发布策略”这一主线
```

### 验收标准
- [ ] 当前工作区所有未提交改动已纳入一个新的本地提交。
- [ ] `git status --short` 在提交后应为空或只剩本次任务包临时文件。
- [ ] 提交信息能覆盖签名修复、workflow 调整与文档/知识库同步。

---

## 2. 方案

### 技术方案
1. 校验当前工作区状态，确认提交范围包含已修改文件与两个 `.helloagents/archive` 目录。
2. 暂存全部当前未提交内容。
3. 使用一条修复型提交信息创建本地提交。
4. 记录提交哈希与摘要，并归档本次方案包。

### 影响范围
```yaml
涉及模块:
  - Git 历史: 新增 1 个本地提交
  - 工作区文件: 当前全部未提交内容
预计变更文件: 当前工作区全部未提交文件与归档目录
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 把多个归档目录一并提交后，后续历史会更杂 | 低 | 已在确认阶段向用户明确提交范围，并按用户选择执行 |
| 提交后若还想拆分，需要额外 amend/rebase | 低 | 本次只做本地提交，后续仍可在推送前调整 |

## 5. 技术决策

### commit-release-signing-fix#D001: 当前全部未提交内容一次性提交
**日期**: 2026-03-12
**状态**: ✅采纳
**背景**: 用户在选择题中明确选择“把当前未提交内容全部一起提交，包括两个归档目录”。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 只提交签名修复相关文件 | 提交更聚焦 | 与用户本轮选择不一致，仍需额外处理归档目录 |

**决策**: 选择方案 B
**理由**: 以用户当前明确选择为准，避免再次拆分工作区。
**影响**: Git 本地提交历史、后续推送/发 tag 的基础提交
