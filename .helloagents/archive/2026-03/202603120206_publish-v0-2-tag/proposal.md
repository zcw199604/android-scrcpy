## 1. 需求

### 背景
当前仓库本地 `main` 相对 `origin/main` 仍领先 1 个提交：`cce9c10 fix: sign GitHub release APK artifacts`。此外，工作区还有两处未提交的 `.helloagents` 归档文件，记录了上一次“release APK 签名修复”提交动作。用户已经明确确认：将这两处归档文件一起提交，然后推送到 SSH 远端 `origin`，再创建并推送 annotated tag `v0.2`，让新的签名修复版本作为正式发布点。

### 目标
- 将当前未提交的 `.helloagents` 归档文件补充为一个独立提交。
- 将本地 `main` 推送到 SSH 远端 `origin/main`。
- 创建并推送 annotated tag `v0.2`。
- 保留本次发布动作的可追溯方案记录与执行日志。

### 约束条件
```yaml
时间约束: 直接基于当前本地 main 执行，不重写已有提交历史
兼容性约束: 不改动既有签名修复提交 cce9c10，仅补充归档文件提交
安全约束: 不使用强制推送，不覆盖已有 tag，仅在本地和远端均不存在 v0.2 时创建标签
业务约束: tag 必须指向最终成功推送到远端 main 的最新提交
```

### 验收标准
- [ ] 当前未提交的 `.helloagents` 归档文件已纳入一个新的本地提交。
- [ ] `main` 已成功推送到 `origin/main`，且远端包含 `cce9c10` 与本次归档补充提交。
- [ ] 本地与远端都存在 annotated tag `v0.2`，且解引用后指向最终发布提交。
- [ ] 执行过程中的关键 Git 状态、风险与结果已写入任务包。

## 2. 方案

### 技术方案
1. 先校验当前工作区范围、分支状态、远端 SSH 地址，以及本地/远端 `v0.2` 是否不存在。
2. 仅暂存并提交本次新增的 `.helloagents` 归档文件，保持与 `cce9c10` 的功能修复提交分离。
3. 执行普通 `git push origin main`，确保远端 `main` 先落到最终发布提交。
4. 在当前 `HEAD` 上创建 annotated tag `v0.2`。
5. 推送 `v0.2` 到 `origin`，并用 `rev-parse` / `ls-remote` 校验远端落点。

### 影响范围
```yaml
涉及模块:
  - Git 历史: 新增 1 个归档补充提交
  - 远端分支: origin/main
  - 远端标签: refs/tags/v0.2
  - repository_docs: 发布记录与归档索引
预计变更文件: 当前两处 `.helloagents` 归档文件 + tag 引用
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 远端 `origin/main` 在推送前发生变化，导致非快进推送失败 | 中 | 推送前先 `fetch` 并确认当前远端位置 |
| 本地或远端已存在同名 `v0.2` 标签 | 中 | 创建前检查本地与远端 tag 是否存在 |
| 把归档文件也纳入发布快照，导致 tag 不再直接指向 `cce9c10` | 低 | 该行为已在确认阶段明确说明，并按用户选择执行 |

## 5. 技术决策

### publish-v0-2-tag#D001: 采用“先推分支、后打标签”的稳妥发布顺序
**日期**: 2026-03-12
**状态**: ✅采纳
**背景**: 用户已经确认执行“提交归档文件 + 推送 main + 打 v0.2 tag”，且任务处于 R3 全自动执行模式，需要在多种执行路径中选择一条默认方案。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 先校验并提交归档文件，推送 main 后再创建并推送 annotated tag | 风险最小，tag 一定落在远端已存在的最终提交上，可追溯性最好 | 步骤略多 |
| B: 补交归档后直接快速发布 | 步骤最少，执行最快 | 前置校验较少，出错窗口更大 |
| C: 强调发布证据链与说明记录 | 审计性最好，记录最完整 | 流程更重，超过当前任务必要复杂度 |
**决策**: 选择方案 A
**理由**: 当前本地相对远端并非零差异，且还有待提交归档文件。采用“最小补充提交 → 推送分支 → 校验远端 → 创建并推送 annotated tag”的顺序，能够避免 tag 指向错误对象，并尽量降低远端发布失败风险。
**影响**: Git 提交历史、`origin/main`、`v0.2` 标签、后续 GitHub tag release 工作流
