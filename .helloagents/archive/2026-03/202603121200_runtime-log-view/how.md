# 技术设计: runtime-log-view

## 目标
在不引入外部依赖、不改变现有连接协议与主流程的前提下，为 Android 主控端增加应用内运行日志查看能力。

## 方案概述
### 方案A（采用）: 进程内环形日志缓冲 + 独立日志 Activity
- 在 `AppData` 中增加线程安全的内存日志缓冲区，统一保存格式化后的日志文本。
- 在 `PublicTools` 中集中封装日志写入：继续输出到 Android `Log`，同时追加到内存缓冲。
- 在 `MainActivity` / `Client` / `ClientStream` 的关键阶段补充信息日志。
- 新增 `LogActivity` 展示当前缓冲区的日志。
- 在设置页增加一个入口跳转到 `LogActivity`。

**优点:**
- 改动集中，易于复用现有 `PublicTools.logToast()`。
- 无需数据库或文件写入，风险低。
- 满足“记录到内存里”的明确要求。

**缺点:**
- 日志只保留在当前进程，应用重启后清空。
- 不能替代系统级 Logcat。

### 方案B（未采用）: 落盘到文件后在应用内读取
**拒绝原因:** 用户明确要求记录到内存里；文件方案会增加权限、I/O、清理策略与隐私边界成本。

## 关键设计
### 1. 日志缓冲区
- 位置: `AppData`
- 数据结构: 有上限的队列（保留最近 N 条）
- 特性:
  - 线程安全
  - 单条日志包含时间、级别、类型和正文
  - 超过上限时自动丢弃最旧日志

### 2. 日志入口统一
- 保持现有 `logToast(type, msg, showToast)` 调用方不变
- 新增 `logInfo(type, msg)` 等轻量入口记录运行阶段日志
- 所有日志先格式化并写入内存，再转发到 Android `Log`

### 3. 关键埋点
- `MainActivity`: 应用启动、自动连接任务触发
- `Client`: 开始连接、连接成功、断开、自动重连
- `ClientStream`: ADB 建连、server 启动、直连/ADB forward 成功、失败、超时、中断
- 现有 `logToast()` 路径继续覆盖 controller/player/server 等错误日志

### 4. 日志查看界面
- 新增 `LogActivity`
- 用 `ScrollView + TextView` 展示内存日志
- 明确提示“仅保存在当前应用进程内存中，重启后清空”
- 打开页面或回到页面时刷新一次日志内容

## 风险与规避
- **风险:** 日志量过大导致内存增长
  - **规避:** 设置固定上限，仅保留最近日志
- **风险:** 多线程写日志导致并发问题
  - **规避:** 在 `AppData` 中使用同步块封装读写
- **风险:** 日志界面展示长文本造成卡顿
  - **规避:** 只展示固定上限内的日志，避免无界累积

## 验证计划
1. 执行 `./gradlew :app:assembleDebug` 验证编译、资源与清单。
2. 静态检查：确认设置页入口顺序正确、Manifest 已注册 `LogActivity`。
3. 静态检查：确认运行日志包含内存缓冲、启动/连接/断开等关键节点。
