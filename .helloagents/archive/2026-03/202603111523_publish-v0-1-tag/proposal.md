## 1. 需求

### 背景
当前工作区已经累积了三类主要改动：移除激活/捐赠门控、清理 README / DONATE / 知识库说明，以及新增 GitHub tag release 打包工作流。用户希望把这些变更作为一次正式发布提交到远端 `origin/main`，并创建、推送注释标签 `v0.1`，让 GitHub Release 工作流能以该 tag 触发。

### 目标
- 将当前工作区的全部改动整理为一次提交。
- 推送到 `origin/main`。
- 创建并推送 annotated tag `v0.1`。
- 保留可追溯的方案记录与执行日志。

### 约束条件
```yaml
时间约束: 直接基于当前工作区执行，不再拆分多个提交
兼容性约束: 不回退已有改动；提交内容以当前工作区事实为准
安全约束: 不使用强制推送；仅在确认远端无同名 tag 时创建并推送 `v0.1`
业务约束: 提交信息需准确概括“开放开源版使用 + 文档清理 + GitHub release workflow”三类改动
```

### 验收标准
- [ ] 当前工作区变更已全部纳入一个新的 Git 提交。
- [ ] `main` 已成功推送到 `origin/main`。
- [ ] 本地与远端都存在 annotated tag `v0.1`，且指向本次新提交。
- [ ] 执行过程中的关键 Git 状态、失败/风险与结果已写入任务包。

## 2. 方案

### 技术方案
1. 先同步远端引用并确认 `main` 与 `origin/main` 没有分叉、`v0.1` 在本地和远端均不存在。
2. 使用单个提交纳入当前工作区改动，提交信息概括三部分：去除激活门控、清理公开文档、补充 GitHub release 打包工作流。
3. 执行普通 `git push origin main`，避免改写远端历史。
4. 创建 annotated tag `v0.1` 并推送到 `origin`。
5. 用 `git rev-parse` / `git ls-remote --tags` 验证分支与标签已经到达远端。

### 影响范围
```yaml
涉及模块:
  - Git 历史: 新增 1 个提交
  - 远端分支: origin/main
  - 远端标签: refs/tags/v0.1
预计变更文件: 当前工作区全部已修改文件 + tag 引用
```

### 风险评估
| 风险 | 等级 | 应对 |
|------|------|------|
| 远端分支在推送前发生变化，导致推送失败 | 中 | 先 fetch 并确认 ahead/behind 为 0/0，再执行 push |
| 远端已存在同名 tag，导致标签推送失败 | 中 | 推送前检查本地/远端 `v0.1` 是否存在 |
| 单提交会把当前工作区全部改动一起发布，后续不便拆分 | 低 | 已在确认阶段向用户明确“提交范围=当前全部改动”，并获得确认 |

## 4. 核心场景

### 场景: 维护者一次性发布当前工作区改动
**模块**: repository_docs
**条件**: `main` 与 `origin/main` 同步，远端不存在 `v0.1`。
**行为**: 维护者将当前工作区的功能、文档与工作流改动作为单个提交推送，并创建 annotated tag `v0.1`。
**结果**: 远端仓库获得可追溯的提交与发布标签，后续 GitHub Release 工作流可据此运行。

## 5. 技术决策

### publish-v0.1-tag#D001: 当前工作区全部改动以单个提交发布
**日期**: 2026-03-11
**状态**: ✅采纳
**背景**: 用户要求“帮我提交推送，tag v0.1”，并在确认阶段接受“单个提交 + annotated tag”的执行方案。
**选项分析**:
| 选项 | 优点 | 缺点 |
|------|------|------|
| A: 按主题拆成多个提交后再打 tag | 历史更细，便于逐项回溯 | 需要重新分拣当前工作区，不符合用户当前“直接发布”诉求 |
| B: 直接把当前工作区改动作为单个提交发布 | 执行快，与用户确认方案一致 | 一个提交中会同时包含代码、文档和 CI 改动 |
**决策**: 选择方案 B
**理由**: 用户已明确确认采用单个提交并直接推送；当前远端和本地同步，可低风险完成发布。
**影响**: Git 提交历史、`origin/main`、`v0.1` 标签
